Prototypes & classes JavaScript

JavaScript inheritance is prototype-based: objects link to other objects. ES6 class is syntactic sugar over this same machinery — understanding the chain underneath is what separates juniors from seniors.

The prototype chain

Every object has an internal [[Prototype]] link. When you read a property, the engine checks the object, then its prototype, then its prototype, up to null. Method lookup walks this chain; the first match wins. Writes, by contrast, happen on the object itself (they don't mutate the prototype).

How method lookup works

Calling arr.map(...) finds map not on the array but on Array.prototype. obj.toString() falls all the way through to Object.prototype.toString. Defining a method on the prototype means one shared copy across all instances (memory-efficient) rather than one per instance.

ES6 class is sugar

A class creates a constructor function; methods declared in the body go on its .prototype. The behaviour is identical to the old constructor-function pattern, just clearer. Classes are not hoisted usefully (TDZ) and run in strict mode.

Inheritance: extends & super

class Dog extends Animal links Dog.prototype to Animal.prototype. In the subclass constructor you must call super(...) before using this; super.method() calls the parent's version.

Object.create

Object.create(proto) makes a new object with its [[Prototype]] set directly to proto — prototypal inheritance without any constructor or class. Object.create(null) makes a truly bare dictionary with no inherited properties (no toString, etc.).

Senior tip: Put shared methods on the prototype (one copy); put per-instance state in the constructor. Classes do this for you automatically — methods land on the prototype, fields on the instance.

🎯 Interview questions

Difference between __proto__ and prototype?

prototype is a property on constructor functions — the object future instances link to. __proto__ is the live link on an instance to its prototype. Thus new Foo().__proto__ === Foo.prototype.

How does property lookup work?

The engine checks the object, then walks the prototype chain via [[Prototype]] until it finds the property or reaches null. First match wins; reads traverse, writes land on the instance.

Is an ES6 class a real class?

No — it's syntactic sugar over constructor functions and prototypes. Methods go on the .prototype; extends wires up the prototype chain.

Why define methods on the prototype instead of in the constructor?

Prototype methods are shared — one function for all instances. Defining them in the constructor creates a separate copy per instance, wasting memory.

What does Object.create(null) give you?

An object with no prototype at all — a clean dictionary with no inherited keys like toString or hasOwnProperty, safe for use as a map.
