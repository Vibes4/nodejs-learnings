Closures JavaScript

A closure is a function bundled together with references to the variables from the scope in which it was created. The function "remembers" that environment even after the outer function has returned.

Lexical scope

JavaScript resolves variable names by where functions are written (lexically), not where they're called. An inner function can always read the bindings of its enclosing functions — that captured chain of scopes is what makes a closure.

Counter / factory example

A factory returns functions that close over private state. Each call to the factory produces an independent environment, so two counters don't interfere.

Data privacy

Before #private class fields, closures were the way to hide state. Variables in the outer scope are unreachable from outside — only the returned functions can touch them. This is the module pattern.

The classic var + setTimeout bug

A loop with var i and an async callback logs the final value of i for every iteration, because all callbacks close over the same binding, which has already advanced by the time they run.

Gotcha: for (var i = 0; i < 3; i++) setTimeout(() => console.log(i)) prints 3 3 3, not 0 1 2.

Two fixes:

  • let — gives each iteration its own binding (modern, idiomatic).
  • IIFE — wrap the body in an immediately-invoked function that captures the current value as a parameter (the pre-ES6 technique).

Memory implications

A closure keeps its captured variables alive as long as the closure itself is reachable. That's usually exactly what you want, but holding a long-lived closure (an event listener, a cache entry) over a large object can prevent garbage collection — a subtle leak. Detach listeners and null out references you no longer need.

Senior tip: Closures power memoization, debounce/throttle, currying, and the entire module pattern. If you can explain the counter factory and the loop bug fluently, you understand closures.

🎯 Interview questions

Define a closure in one sentence.

A function together with the lexical environment (the variables) it was defined in, which it retains access to even after the outer function returns.

Why does the var + setTimeout loop print the same number?

All callbacks share one var binding. By the time the timers fire, the loop has finished and that single variable holds its final value.

How do you fix it without let?

Wrap the iteration in an IIFE that takes i as a parameter, creating a new scope and capturing the current value per iteration.

How do closures enable data privacy?

State declared in the outer function is unreachable from outside; only the returned inner functions can read or mutate it. This is the module pattern.

Can closures cause memory leaks?

Yes — a reachable closure keeps its captured variables alive. Long-lived closures (listeners, caches) over big objects can block GC until you remove the reference.
