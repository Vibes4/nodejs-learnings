Promises & async/await JavaScript

Promises model an eventual value; async/await is syntax that lets you write asynchronous code that reads like synchronous code. Mastering ordering and error handling here is essential for any backend role.

Promise states

  • pending — not yet settled.
  • fulfilled — resolved with a value.
  • rejected — failed with a reason.

A promise settles once and is then immutable. async functions always return a promise; await unwraps one (and rethrows rejections).

Chaining

Each .then() returns a new promise, so they chain. Returning a value passes it on; returning a promise waits for it. A thrown error or rejection skips down to the next .catch().

Error handling

  • With await: wrap in try/catch.
  • With chains: end with .catch(); one handler covers every prior step.
  • .finally() runs regardless of outcome (cleanup).
Gotcha: A rejected promise with no handler triggers an unhandledRejection — in modern Node this crashes the process by default. Always attach a catch.

Microtask vs macrotask ordering

Promise callbacks are microtasks; setTimeout/setImmediate are macrotasks. After the current synchronous code, the engine drains the entire microtask queue before running the next macrotask. So a resolved promise's .then always runs before a setTimeout(0) scheduled at the same time.

await-in-loop pitfall

Awaiting inside a for loop runs requests sequentially — each waits for the previous. If they're independent, kick them all off and await together with Promise.all for concurrency. Use sequential await only when each step depends on the previous result or you must rate-limit.

Senior tip: Promise.all fast-fails on the first rejection; if you need every result regardless of failures (e.g. a dashboard of independent calls), use Promise.allSettled.

🎯 Interview questions

What are the three promise states?

pending, fulfilled, rejected. A promise settles exactly once and then never changes.

Why does a promise .then run before setTimeout(0)?

Promise callbacks are microtasks; timers are macrotasks. The whole microtask queue drains after the current task and before the next macrotask.

Difference between Promise.all and allSettled?

all rejects as soon as any input rejects (and you lose the other results). allSettled always fulfils with a status object per input, so partial failures are visible.

What's wrong with awaiting in a loop?

It serializes independent async work. For concurrency, start all promises and await Promise.all([...]). Sequential await is only correct when steps depend on each other.

What happens to an unhandled rejection in Node?

It fires the unhandledRejection event and, by default in modern Node, terminates the process. Always attach .catch or wrap awaits in try/catch.
