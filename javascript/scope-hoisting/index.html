Scope & hoisting JavaScript

Scope determines where a binding is visible. Hoisting describes how declarations are processed before any code runs. Mixing up var, let, and const behaviour here is a classic interview trap.

Function scope vs block scope

var ignores { } blocks (if/for/while) — it's scoped to the enclosing function. let/const respect every block. A var declared inside an if leaks to the whole function; a let does not.

Hoisting

  • Function declarations are hoisted entirely — you can call them before they appear in source.
  • var declarations are hoisted and initialised to undefined (the assignment stays in place).
  • let/const declarations are hoisted but not initialised — accessing them before the declaration throws.
  • Function expressions / arrow functions follow their variable's rules (so a const fn = () => {} is in the TDZ before its line).

The Temporal Dead Zone (TDZ)

The region from the start of a block until a let/const declaration is the TDZ. Reading the variable there throws ReferenceError: Cannot access 'x' before initialization — distinct from the silent undefined you'd get from var. The TDZ exists to make const-before-use a hard error.

let/const in loops

A let in a for header creates a fresh binding per iteration. That's why closures (e.g. setTimeout callbacks) capture the correct value with let but all see the final value with var (a single shared binding).

Senior tip: Default to const, reach for let only when you must reassign, and never use var in new code. This eliminates an entire class of scope and hoisting bugs.

🎯 Interview questions

What is the TDZ?

The span between entering a scope and the let/const declaration line. Accessing the binding in that span throws a ReferenceError rather than returning undefined.

Are let and const hoisted?

Yes — the engine knows about them at the top of the block — but unlike var they aren't initialised, so they're unusable until their declaration runs (the TDZ).

Why does let fix the for-loop closure bug?

Each iteration gets its own binding, so each closure captures a distinct value. var has one binding shared across all iterations.

Does const make an object immutable?

No. It only prevents reassigning the variable. Use Object.freeze() for a shallow immutable object.

Difference between a function declaration and expression w.r.t. hoisting?

Declarations are fully hoisted and callable before their line. Expressions assigned to a variable follow that variable's hoisting rules (undefined for var, TDZ for let/const).
