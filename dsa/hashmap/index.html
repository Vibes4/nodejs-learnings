HashMap / HashSet DSA

Note: This is the single most important topic in coding interviews. A huge fraction of array/string problems collapse from O(n²) to O(n) the moment you introduce a hash map.

A HashMap stores key→value with O(1) average access by hashing the key to a bucket. A HashSet is the same idea for membership only. In JS: Map and Set (or plain objects for string/number keys).

When to reach for it

  • Frequency counting — how many times each item appears.
  • Fast lookup — "have I seen X?" in O(1).
  • Duplicate detection — Set membership while iterating.
  • Complement / pairing — store what you need, check on arrival (Two Sum).

Complexity

OperationAverageWorst
GetO(1)O(n) (collisions)

Has / deleteO(1)O(n)

Common questions

  • Two Sum — complement lookup.
  • Contains Duplicate — Set membership in one pass.
  • Valid Anagram — character frequency map.
  • Frequency Counter — most common element, group by count.
Gotcha: Use Map, not a plain object, when keys aren't strings/numbers, when you need insertion order guaranteed, or when keys might collide with prototype names like "constructor".

🎯 Interview questions

When do you reach for a HashMap?

Frequency counting, O(1) membership/lookup, duplicate detection, and complement/pairing problems — anytime you'd otherwise nest a loop to "find a match."

Map vs plain object in JS?

Map allows any key type, preserves insertion order, has a .size, and avoids prototype-key collisions. Objects are fine for simple string-keyed counters and are slightly faster for that case.

Why is average O(1) but worst O(n)?

Hashing spreads keys across buckets for O(1) average; pathological collisions (or a bad hash) put everything in one bucket, degrading to a linear scan.

Set vs Map — when?

Set when you only care about presence (dedupe, "seen"); Map when you need an associated value (index, count, node).
