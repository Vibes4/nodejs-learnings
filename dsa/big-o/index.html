Big-O Cheat Sheet DSA

Big-O describes how an algorithm's cost grows as input size n grows. It drops constants and lower-order terms — we care about the shape of the curve, not the exact wall-clock time.

Complexity table

Big-ONameExample
O(1)ConstantHash lookup, array index
O(log n)LogarithmicBinary search
O(n)LinearSingle loop over input
O(n log n)LinearithmicMerge sort, heap sort
O(n²)QuadraticNested loops, naive pair check
O(2ⁿ)ExponentialBacktracking, naive recursion

Time vs space

Time = number of operations. Space = extra memory (the auxiliary memory, not the input itself). A HashMap solution to Two Sum trades O(n) space to cut time from O(n²) to O(n) — a classic time/space trade-off.

Best / average / worst

  • Worst case — the guarantee you quote in interviews unless asked otherwise.
  • Average case — expected over typical inputs (e.g. quicksort is O(n log n) average, O(n²) worst).
  • Best case — rarely useful (e.g. already-sorted input).

Amortized analysis

A dynamic array push is O(1) amortized: most pushes are O(1), but occasionally the array doubles and copies all elements (O(n)). Spread across all pushes, the average is still O(1).

Pattern: When you see a nested loop, ask "can a HashMap or sorting collapse this to O(n) or O(n log n)?"
Gotcha: O(n + m) with two different inputs is NOT O(2n) = O(n). Keep distinct variables distinct.

🎯 Interview questions

What does Big-O actually measure?

The asymptotic growth rate of an algorithm's resource use (time or space) as input size grows, ignoring constants and lower-order terms.

Difference between average and amortized?

Average is a probabilistic claim over inputs. Amortized is a guarantee over a sequence of operations — every sequence of n pushes costs O(n) total, regardless of input distribution.

Why is binary search O(log n)?

Each step halves the search space, so you need log₂(n) steps to reach a single element.

When is O(n²) acceptable?

For small or bounded n (e.g. n < 1000), or when the constant factor and code simplicity outweigh scaling concerns.

Does dropping constants ever mislead?

Yes — for small n a "slower" Big-O can be faster in practice. Big-O describes scaling, not absolute speed.
