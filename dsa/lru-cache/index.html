LRU Cache DSA

A Least-Recently-Used cache holds a fixed number of entries; when full, it evicts the entry that hasn't been touched for the longest time. It's a classic design question because it combines two structures to hit O(1) on both operations.

The design

HashMap + Doubly Linked List.

  • HashMap — key → node, giving O(1) lookup.
  • Doubly Linked List — maintains usage order. Most-recently-used at one end (head), least-recently-used at the other (tail).

On get or put, move the node to the head (most recent). On overflow, evict the tail. A doubly linked list makes both unlink and move-to-head O(1) because each node knows its neighbors.

Complexity

OperationBig-O
GetO(1)
PutO(1)
SpaceO(capacity)
Pattern: "O(1) get and O(1) put with eviction" is the signature of HashMap + Doubly Linked List. Sentinel head/tail nodes remove every null-check edge case.
Gotcha: A singly linked list can't do O(1) removal of an arbitrary node (you'd need the previous node). That's why the doubly linked list is essential. In JS, an insertion-ordered Map can fake it: delete-then-set moves a key to the end.

🎯 Interview questions

Why HashMap + doubly linked list?

The map gives O(1) key lookup; the doubly linked list gives O(1) reordering and eviction. Together both get and put are O(1).

Why not a singly linked list?

Removing or moving an arbitrary node needs its predecessor. A singly linked list would require an O(n) scan to find it; doubly linked nodes carry a prev pointer.

What gets evicted and when?

When inserting into a full cache, the least-recently-used entry (the list tail) is removed before adding the new entry at the head.

Can you implement it with a JS Map?

Yes — Map preserves insertion order, so on access you delete and re-set the key to move it to the "newest" position, and evict the first key (oldest) when over capacity.
