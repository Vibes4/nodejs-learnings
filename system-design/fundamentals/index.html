System design fundamentals System Design

System design interviews test whether you can take a vague prompt ("design Twitter") and reason about scale, tradeoffs, and bottlenecks out loud. There is no single right answer — interviewers grade your structured thinking.

Scaling: vertical vs horizontal

Vertical (scale up)Bigger machine (more CPU/RAM). Simple, no code changes. Hard ceiling, single point of failure, expensive at the top end.
Horizontal (scale out)More machines behind a load balancer. Near-unlimited, fault-tolerant. Needs statelessness, coordination, and a distributed data story.

Real systems do both, but interviews almost always want horizontal scaling — say so early.

Load balancing

A load balancer (L4 TCP or L7 HTTP) spreads traffic across a pool. Strategies: round-robin, least-connections, consistent hashing (sticky to a node, minimizes reshuffling when nodes change). It also does health checks and removes dead nodes.

Statelessness

If any server can handle any request, you can add/remove servers freely. Push state out: sessions to Redis, files to object storage, data to a shared DB. Sticky sessions are a smell — they break clean horizontal scaling.

CAP theorem

In the presence of a network Partition, you must choose between Consistency (every read sees the latest write) and Availability (every request gets a non-error response). You cannot have both during a partition.

  • CP — reject/block until consistent (e.g. a bank ledger, etcd, ZooKeeper).
  • AP — serve possibly-stale data, reconcile later (e.g. DNS, Cassandra, DynamoDB default).
Gotcha: Partitions are not optional — networks fail. So the real choice is CP vs AP. "CA" only describes a single node with no partitions.

Consistency models

  • Strong: after a write completes, all reads see it. Easier to reason about, costs latency/availability.
  • Eventual: replicas converge "eventually". Cheap, highly available; clients may read stale data. Fine for likes/feeds, not for balances.

Latency vs throughput

Latency = time for one request (p50/p95/p99 — quote percentiles, not averages). Throughput = requests per second the system sustains. You optimize them differently: batching/queuing raises throughput but can raise latency.

SQL vs NoSQL

SQL (Postgres/MySQL)Strong schema, ACID transactions, joins, ad-hoc queries. Default choice. Vertical scaling + read replicas; sharding is manual.
NoSQL (Cassandra/Dynamo/Mongo)Flexible schema, horizontal scale built-in, high write throughput, eventual consistency. Weak/limited joins & transactions. Choose for huge scale or simple access patterns.

Replication & sharding

  • Replication = copies of the same data (leader/follower). Improves read throughput & availability; writes go to the leader.
  • Sharding (partitioning) = splitting data across nodes by a key (hash or range). Scales writes & storage; introduces cross-shard query pain and hot-shard risk (pick a good shard key).

A structured interview approach

  1. Requirements — functional + non-functional (scale, latency, consistency). Clarify before designing.
  2. Estimates — back-of-envelope QPS, storage, bandwidth.
  3. API — define the key endpoints / contracts.
  4. Data model — entities, SQL vs NoSQL, schema.
  5. High-level design — boxes & arrows: LB, services, cache, DB, queue.
  6. Scale & bottlenecks — cache, shard, replicate, async, CDN; discuss failure modes.
Senior tip: Drive the conversation. State assumptions, name the bottleneck before the interviewer does, and always frame choices as tradeoffs ("I'd pick AP here because stale likes are acceptable but downtime isn't").

🎯 Interview questions

Explain CAP and which side you'd pick for a shopping cart.

Under a partition you choose consistency or availability. For a cart I lean AP — keep it editable and merge conflicting versions later (Dynamo-style) — but checkout/payment must be CP so you never double-charge.

Difference between replication and sharding?

Replication copies the same dataset for read scaling & HA; writes still hit one leader. Sharding splits different data across nodes to scale writes/storage. They're orthogonal and usually combined.

Why prefer percentiles over average latency?

Averages hide tail latency. A p99 of 2s means 1% of users have a terrible experience even if the mean is 50ms; SLAs are written against p95/p99.

When would you choose NoSQL over SQL?

Massive horizontal write scale, simple key-based access, flexible/evolving schema, and tolerance for eventual consistency. If you need joins, transactions, or ad-hoc queries, start with SQL.

How do you make a service horizontally scalable?

Make it stateless, put it behind a load balancer, externalize session/state to Redis & object storage, and ensure the data layer can replicate/shard. Then adding nodes is free.
