URL shortener System Design

Design a service that turns https://example.com/very/long/path into https://sho.rt/aZ4k9 and redirects on lookup. A classic interview because it surfaces hashing, encoding, read-heavy scaling, and caching.

Requirements

  • Functional: create a short URL for a long one; redirect short → long; optional custom alias; optional expiry; basic analytics (click counts).
  • Non-functional: very low-latency redirects, extremely read-heavy (~100:1), highly available, short keys.

Estimating key length (base62)

Use base62 = [0-9a-zA-Z] = 62 symbols. A length-7 key gives 62⁷ ≈ 3.5 trillion combinations — plenty. Base62 is URL-safe and compact (shorter than hex or base10 ids).

Generating short keys

Counter + base62Maintain a global auto-increment id, encode it to base62. No collisions ever, keys are short and sequential. Need a distributed counter (e.g. range allocation / ticket server).
Hash (MD5/SHA) + truncateHash the long URL, take first N chars. Same URL → same key (free dedupe). But truncation causes collisions — must check & retry.
Key Generation Service (KGS)A standalone service pre-generates random unique keys into a "available" pool; servers grab keys in batches. Decouples generation from write path, no collisions at request time.

Collision handling

For hash-based keys: on insert, rely on a unique constraint; on conflict, append a salt / re-hash and retry. Counter and KGS approaches avoid collisions by construction.

Read-heavy: cache aggressively

Redirects vastly outnumber creations. Put a cache (Redis) + CDN in front of the lookup; the hot set of popular links lives in memory. The DB only sees misses and writes.

Data model

A simple key-value table works: short_key (PK), long_url, created_at, expires_at, click_count. Sharded by short_key hash at scale. NoSQL/KV stores fit well given the access pattern.

Redirect: 301 vs 302

  • 301 (permanent): browsers cache it, so subsequent clicks skip your server — great for load, but you lose click analytics and can't change the target.
  • 302 (temporary): not cached; every click hits your server, enabling analytics and updatable targets. Usually the right choice for a shortener.
Gotcha: Choosing 301 silently kills your analytics because the browser stops asking your server. Most shorteners use 302 deliberately.

Analytics

Don't update the DB synchronously on every click — that adds write load to the hot redirect path. Fire an async event (queue/log) and aggregate click counts, geo, and referrers offline.

Senior tip: Counter+base62 with a ticket/range allocator gives the shortest keys and zero collisions; reach for hashing only if you specifically want identical-URL dedupe.

🎯 Interview questions

How do you generate short, unique keys?

Best: a global auto-increment id encoded to base62 — short, collision-free. Alternatives: hash-and-truncate (needs collision retries, gives dedupe) or a Key Generation Service that pre-mints unique keys into a pool.

Why base62 and what length?

Base62 is URL-safe and compact. Length 7 yields 62⁷ ≈ 3.5 trillion keys, enough for years of growth while staying short.

301 or 302 redirect?

302. A 301 is cached by browsers so clicks bypass your server, killing analytics and the ability to change the target. 302 keeps every click flowing through you.

How do you scale the read path?

It's ~100:1 read-heavy, so cache aggressively: Redis for hot keys plus a CDN; the DB handles only misses and writes. Shard the store by short_key.

How do you generate unique ids across many servers?

Hand each server a contiguous id range from a central allocator (or use Snowflake-style ids) so they never coordinate per request yet never collide.
