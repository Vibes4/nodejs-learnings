Caching & Redis System Design

A cache stores expensive-to-compute or slow-to-fetch data in fast memory close to the consumer. It's the single highest-leverage tool for read-heavy systems — but cache correctness is genuinely hard.

Why cache?

  • Cut latency (RAM in microseconds vs disk/DB in milliseconds).
  • Reduce load on the database so it can scale.
  • Smooth out spikes and absorb read bursts.

Caches live at many layers: browser, CDN, reverse proxy, application (Redis/Memcached), and inside the DB itself.

Caching patterns

Cache-aside (lazy)App checks cache; on miss, reads DB and populates cache. Most common. Cache only holds requested data. Risk: first request is slow; stale data until TTL.
Read-throughApp talks only to the cache; the cache library loads from DB on miss. Cleaner app code; needs cache that supports it.
Write-throughWrites go to cache and DB synchronously. Cache always fresh; writes are slower.
Write-back (write-behind)Write to cache, flush to DB asynchronously. Fast writes, absorbs bursts. Risk: data loss if cache dies before flush.

Eviction policies

  • LRU — evict least-recently-used. Good default; matches temporal locality.
  • LFU — evict least-frequently-used. Better for stable hot sets, worse for changing workloads.
  • TTL — expire after a fixed time. Bounds staleness regardless of access.

Cache invalidation — the hard problem

"There are only two hard things in CS: cache invalidation and naming things." When the source data changes, the cache must too. Options:

  • TTL — simplest; accept staleness up to the TTL.
  • Write-through / explicit delete — delete or update the key on every write.
  • Versioned keys — embed a version in the key so old data is unreachable.
Gotcha: Prefer deleting the key on write over updating it — concurrent writes can interleave and leave the cache holding the wrong value.

Thundering herd / cache stampede

When a hot key expires, thousands of concurrent requests all miss and hammer the DB at once. Mitigations:

  • Locking / single-flight — only one request recomputes; others wait.
  • Stale-while-revalidate — serve stale data while one worker refreshes.
  • Jittered TTLs — randomize expiry so keys don't all die together.

Redis use cases

CacheKey-value with TTL; the default.
Session storeExternalized sessions for stateless app servers.
Rate limitingAtomic INCR + EXPIRE counters.
LeaderboardsSorted sets (ZADD/ZRANGE) give ranked queries for free.
Pub/Sub & queuesLightweight messaging, streams, job queues.
Senior tip: Redis is single-threaded for commands, so multi-step operations are best done atomically with Lua scripts or MULTI/EXEC to avoid races (e.g. token-bucket rate limiting).

🎯 Interview questions

Cache-aside vs write-through — when each?

Cache-aside for read-heavy data where occasional staleness is fine (the default). Write-through when reads must always see the latest write and you can tolerate slower writes. Write-back when write throughput matters more than durability.

How do you prevent a cache stampede?

Single-flight locking so only one request recomputes a hot key, serve stale-while-revalidate, and add jitter to TTLs so keys don't expire in lockstep.

Update or delete the cache key on a DB write?

Delete. Updating can race with concurrent writes and leave a stale value; deleting forces the next read to repopulate from the source of truth.

What makes Redis fast and what's the catch?

In-memory + single-threaded event loop means no lock contention and predictable latency. The catch: a slow command blocks everything, and memory is finite — you need eviction policies and persistence (RDB/AOF) for durability.

How would you build a leaderboard?

A Redis sorted set: ZADD board score user, then ZREVRANGE for top-N and ZRANK for a user's position — all O(log n), no scanning.
