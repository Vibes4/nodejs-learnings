File upload service System Design

Design a service for uploading and serving files (images, videos, documents) — the kind backing S3-style storage, photo apps, or document platforms.

Direct vs presigned-URL upload

Direct (through your server)Client POSTs to your API, which streams to object storage. Simple, but your servers carry all the bandwidth — a scaling bottleneck.
Presigned URLClient asks your API for a short-lived signed URL, then uploads directly to object storage (S3). Offloads bandwidth from your servers; your API only handles metadata. The standard at scale.
Senior tip: Always reach for presigned URLs. Your app servers should handle metadata and authorization, not gigabytes of file bytes.

Large files: multipart / chunked upload

Split big files into chunks (e.g. 5–10 MB) uploaded independently, then assembled. Benefits:

  • Parallelism — upload chunks concurrently for speed.
  • Resumability — a dropped connection only re-sends the failed chunk, not the whole file.
  • Reliability — each chunk is acked; the server tracks which are received and reassembles when complete.

Validation & virus scanning

Validate type/size server-side (don't trust the client's content-type or extension — check magic bytes). Quarantine uploads, run async virus/malware scanning, and only mark a file "available" after it passes. Never serve unscanned user content from a trusted domain.

Gotcha: Client-side validation is for UX only. A determined client bypasses it — re-validate size, type, and content on the server.

CDN for downloads

Serve reads through a CDN (CloudFront) so files are cached at edge locations near users — lower latency, less origin load. Use signed CDN URLs / tokens for access control on private files.

Metadata DB

Object storage holds the bytes; a database holds the metadata: file id, owner, name, size, content type, content hash, storage key, status (uploading/scanning/available), timestamps. This is what you query and authorize against.

Deduplication by content hash

Hash the content (SHA-256). If the hash already exists in storage, skip the upload and just point a new metadata row at the existing object — saving storage and bandwidth. This also enables integrity checks and resumable-chunk verification.

Gotcha: Content-hash dedupe means deleting a file must be reference-counted — don't delete the underlying object while another row still references the same hash.

🎯 Interview questions

Why presigned URLs instead of uploading through your API?

They let the client upload bytes straight to object storage, so your app servers don't become a bandwidth bottleneck. Your API only issues short-lived signed URLs and records metadata.

How do you support huge files and resumability?

Multipart/chunked upload: split into chunks uploaded independently and in parallel, ack each, and reassemble. A failure re-sends only the missing chunk, not the whole file.

How does content-hash dedupe work, and what's the catch?

Hash the content (SHA-256); if it already exists, point a new metadata row at the existing object instead of re-storing. The catch: deletes must be reference-counted so you don't drop an object another row still uses.

How do you handle malicious uploads?

Validate type/size server-side via magic bytes, quarantine the file, scan asynchronously for malware, and only flip status to "available" after it passes. Serve via signed URLs from a separate domain.

How do you scale downloads?

Put a CDN in front of object storage to cache files at the edge, cutting latency and origin load; use signed CDN URLs for access control on private content.
