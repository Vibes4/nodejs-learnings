Chat system System Design

Design a real-time messaging service (think WhatsApp/Slack). The core challenges are persistent connections at scale, fan-out, presence, and ordering.

Requirements

  • Functional: 1:1 and group chat, online presence, delivery receipts (sent/delivered/read), message history, push for offline users.
  • Non-functional: low latency, high availability, ordered delivery, huge concurrent connection count.

Transport: WebSockets vs long-polling

WebSocketFull-duplex persistent TCP connection; the server can push instantly. The standard choice for chat.
Long-pollingClient holds an HTTP request open until there's data, then reconnects. Works through old proxies; higher overhead and latency. A fallback.
SSEServer-sent events: one-way server→client push. Simpler than WS but no client→server channel.

Connection servers + session store

Users connect to one of many stateless-ish connection (gateway) servers that each hold thousands of live WebSockets. Because a recipient may be connected to a different server, you need a session store / service discovery mapping userId → connectionServer (in Redis). To deliver a message, look up the recipient's server and route to it (often via an internal pub/sub bus).

Message fan-out

  • 1:1: look up the recipient's connection server and push.
  • Group: fan out to each member. Small groups → push directly; large groups/channels → a pub/sub topic per room that connection servers subscribe to.

Storing messages

Persist every message for history and offline delivery, partitioned by chat/conversation id and ordered by time. A wide-column store (Cassandra) or sharded SQL fits the high write volume and time-ordered reads.

Presence

Track online/offline via connection lifecycle + heartbeats. Store userId → lastSeen in Redis with a short TTL refreshed by pings; missing/expired = offline. At scale, presence updates are themselves fanned out only to interested contacts to avoid a broadcast storm.

Offline users & push

If the recipient isn't connected, store the message as undelivered and trigger a push notification (APNs/FCM) via the notification service. On reconnect, the client syncs missed messages by last-seen message id.

Ordering

Guarantee order per conversation, not globally. Use a monotonic per-conversation sequence number or server timestamps; clients sort by it and dedupe by message id (delivery is at-least-once).

Gotcha: Don't rely on client clocks for ordering — they drift. Assign a server-side sequence/timestamp per conversation.
Senior tip: The hard part isn't the chat logic — it's routing a message to a recipient who's connected to a different gateway. Solve it with a Redis session map plus an internal pub/sub bus between connection servers.

🎯 Interview questions

WebSockets vs long-polling for chat?

WebSockets: a persistent full-duplex connection lets the server push instantly with low overhead — the default. Long-polling is a fallback for restrictive networks but has more latency and reconnect overhead.

How do two users on different servers exchange messages?

A Redis session store maps userId→gateway server. The sender's server looks up the recipient's server and routes the message there (typically over an internal pub/sub bus), which pushes it down the recipient's socket.

How do you handle group/channel fan-out at scale?

Use a pub/sub topic per room. Connection servers subscribe on behalf of their connected members, so one publish fans out without the sender knowing every recipient's location.

How is presence implemented?

Connection lifecycle plus heartbeats: store userId→lastSeen in Redis with a short TTL refreshed by pings. Expired = offline. Fan presence changes only to interested contacts to avoid broadcast storms.

How do you guarantee ordering?

Order per conversation using a server-assigned monotonic sequence number or timestamp; clients sort by it and dedupe by message id, since delivery is at-least-once.
