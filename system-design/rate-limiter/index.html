Rate limiter System Design

A rate limiter caps how many requests a client may make in a window. It protects services from abuse, accidental floods, and runaway costs, and enforces fair usage / billing tiers.

Why & where

  • Why: prevent DoS, protect downstreams, ensure fairness, cap cost.
  • Where: usually at the API gateway / edge (before requests hit your services), keyed by API key, user id, or IP. Can also be per-endpoint or per-tenant.

Return 429 Too Many Requests with a Retry-After header, and expose X-RateLimit-Limit / X-RateLimit-Remaining.

Algorithms

AlgorithmHow it worksTradeoff
Fixed windowCount requests per fixed interval (e.g. per minute), reset at boundary.Simple & cheap. Burst at edges: 2x limit possible across a boundary.
Sliding window logStore a timestamp per request; count those within the last window.Perfectly accurate, smooth. Memory-heavy (one entry per request).
Sliding window counterWeighted blend of current + previous fixed window counts.Approximate but smooth; low memory. Great practical default.
Token bucketBucket refills tokens at a steady rate; each request spends one.Allows controlled bursts up to bucket size; smooth average. Very common.
Leaky bucketRequests queue and drain at a fixed rate (like a leaking bucket).Smooths output to a constant rate; no bursts; adds queuing latency.
Gotcha: Fixed window lets a client send limit requests at 0:59 and another limit at 1:00 — effectively 2x the limit in 2 seconds. Sliding window or token bucket fixes this.

Token bucket — the workhorse

Two parameters: capacity (max burst) and refill rate (sustained throughput). It permits short bursts while bounding the long-run average — closest to how real traffic behaves.

Distributed rate limiting

With many gateway nodes, a local counter under-counts. Centralize state in Redis: INCR a per-key counter with EXPIRE, or run a token-bucket Lua script atomically so refill+consume is race-free. Tradeoffs:

  • Centralized (Redis): accurate, but adds a network hop and a shared dependency.
  • Local + sync: each node limits locally and periodically reconciles — lower latency, approximate.
Senior tip: Run the whole check-and-decrement as a single atomic Redis Lua script. A read-modify-write across two round-trips races under load and silently lets clients exceed the limit.

🎯 Interview questions

Token bucket vs leaky bucket?

Token bucket allows bursts up to capacity then enforces an average rate — good for APIs. Leaky bucket drains at a constant rate and queues the rest — good when the downstream needs a perfectly smooth, constant feed.

Why is fixed window problematic?

Boundary bursting: a client can fire a full limit at the end of one window and a full limit at the start of the next, doubling the effective rate over a short span. Sliding window or token bucket avoids this.

How do you rate-limit across many servers?

Centralize counters in Redis with atomic operations (Lua script or INCR+EXPIRE) keyed by user/IP. Optionally use local limiting with periodic sync to cut latency at the cost of exactness.

What response and headers do you return when limited?

HTTP 429 with Retry-After, plus X-RateLimit-Limit, X-RateLimit-Remaining, and a reset timestamp so well-behaved clients can back off.

What key do you rate-limit on?

Depends on the threat: API key/user id for fairness & billing, IP for anonymous abuse (watch out for shared NAT/proxies), and per-endpoint for protecting expensive routes.
