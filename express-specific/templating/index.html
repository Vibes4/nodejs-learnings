Template engines (SSR) Express

Template engines render dynamic HTML on the server. Express supports many — EJS, Pug, Handlebars — via res.render().

Setup

app.set('view engine', 'ejs');     // engine
app.set('views', path.join(__dirname, 'views'));  // template folder

app.get('/', (req, res) => {
  res.render('home', { title: 'Hi', users: [...] });  // renders views/home.ejs
});

When SSR matters

  • SEO-critical pages (server sends fully rendered HTML).
  • Simple admin panels, emails, server-rendered marketing pages.
  • Faster first paint than a client-rendered SPA.
Modern reality: Most apps today are API + React/Vue SPA, or use a meta-framework (Next.js, Remix) for SSR. Classic Express templating is still common for admin tools, server-rendered emails, and small apps.
XSS: Use escaped output (<%= %> in EJS) for user data. Unescaped (<%- %>) injects raw HTML — only for trusted content.

🎯 Interview questions

SSR vs CSR vs SPA?

SSR renders HTML on the server (better SEO/first paint). CSR/SPA renders in the browser via JS (richer interactivity). Many apps mix both.

How do template engines prevent XSS?

By auto-escaping interpolated values. You must opt into raw HTML explicitly, and only for trusted data.

Where does Express look for views?

The directory set via app.set('views', ...) (default ./views), with the configured view engine extension.
