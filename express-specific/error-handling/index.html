Error Handling in Express Express

Express routes errors through its middleware pipeline. The key is the error-handling middleware, identified by its four arguments (err, req, res, next). You can attach handlers at several levels — application, router, and route.

The middleware levels (where handlers live)

  • Application-level — app.use(...) / app.METHOD(...): runs for the whole app.
  • Router-level — router.use(...): scoped to one express.Router().
  • Route-level — handlers passed directly to a route.
  • Error-handling — the special 4-arg signature; can be app- or router-scoped.
  • Built-in / third-party — express.json(), helmet, etc. (can throw → caught by error handlers).

How an error reaches a handler

  • Throw synchronously in a handler → Express catches it.
  • Call next(err) with any argument → skips straight to the next error handler.
  • Express 4: rejected promises are NOT auto-caught — wrap async handlers. Express 5: they are.

1. Route-level error (sync throw)

app.get('/sync', (req, res) => {
  throw new AppError('bad thing', 400);   // Express forwards this to the error handler
});

2. Route-level async error → next(err) via wrapper

const asyncHandler = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);   // rejection -> next(err)

app.get('/async', asyncHandler(async (req, res) => {
  const user = await db.findUser();      // if this rejects, error handler runs
  if (!user) throw new AppError('not found', 404);
  res.json(user);
}));

3. Router-level error handler (scoped)

const users = express.Router();
users.get('/:id', (req, res) => { throw new AppError('boom', 500); });

// 4-arg handler attached to THIS router only
users.use((err, req, res, next) => {
  res.status(err.code || 500).json({ scope: 'users-router', error: err.message });
});
app.use('/users', users);

4. 404 catch-all (after all routes)

app.use((req, res) => res.status(404).json({ error: `Not found: ${req.path}` }));

5. Application-level (global) error handler — always LAST

app.use((err, req, res, next) => {
  console.error(err);                              // full detail server-side
  const status = err.code || 500;
  res.status(status).json({
    error: err.isOperational ? err.message : 'Internal server error'  // sanitize unknowns
  });
});
Order matters. Error handlers must be registered after the routes they protect. A router-level handler catches errors from that router first; anything it re-passes with next(err) bubbles up to the app-level handler.
Never leak internals: log the stack server-side, return a sanitized message + status to the client. Map known AppErrors to status codes; everything else → generic 500.

🎯 Interview questions

How does Express recognize an error handler?

By arity — exactly four parameters (err, req, res, next). Three-arg functions are normal middleware.

Difference between app-level and router-level handlers?

App-level (app.use) applies globally; router-level (router.use) is scoped to that router. A router handler can handle locally or next(err) to bubble up to the global one.

Are async errors caught automatically?

Not in Express 4 — wrap with a helper or try/catch + next(err). Express 5 forwards rejected promises automatically.

How do you handle 404s vs thrown errors?

A normal middleware after all routes returns 404 for unmatched paths; the 4-arg middleware handles thrown/next(err) errors.

Where do body-parser / validation errors go?

They call next(err) internally, so they land in your error-handling middleware just like route errors.
