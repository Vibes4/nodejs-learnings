Validation & security Express

Never trust client input. Validation + a few security middlewares cover most of the OWASP basics interviewers probe.

Input validation

  • express-validator — middleware-style validation/sanitization.
  • Joi / Zod — schema-based validation (Zod is TS-first and popular).
  • Validate types, ranges, formats; sanitize (trim, escape); reject unknown fields.

Security middleware

  • helmet — sets safe HTTP headers (CSP, HSTS, X-Frame-Options, etc.).
  • cors — controls cross-origin access; never blindly use origin: '*' with credentials.
  • express-rate-limit — throttle requests (brute-force / DoS protection).
  • hpp — HTTP parameter pollution; express-mongo-sanitize — strip $/. operators.

Common vulnerabilities to mention

  • SQL/NoSQL injection — use parameterized queries / sanitize operators.
  • XSS — escape output, set CSP.
  • CSRF — same-site cookies / CSRF tokens for cookie-based auth.
  • Secrets — in env vars, never in code; rotate keys.
  • Disable x-powered-by; keep deps patched (npm audit).
Validate at the boundary. Reject bad input early with 400/422 before it touches your DB or business logic. Whitelist, don't blacklist.

🎯 Interview questions

How do you prevent NoSQL injection in Mongo?

Validate/cast input types, avoid passing raw objects as query operators, use express-mongo-sanitize to strip $/., and use schema validation.

What does helmet do?

Sets a bundle of security-related HTTP headers (CSP, HSTS, no-sniff, frameguard) with sensible defaults.

CORS in one sentence?

A browser security mechanism; the server uses CORS headers to declare which origins may call it. Configure explicitly; avoid * with credentials.

Where should validation live?

At the API boundary as middleware, before controllers — plus DB-level constraints as defense in depth.
