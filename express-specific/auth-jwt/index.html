Authentication (JWT) Express

Authentication = who you are; authorization = what you can do. The two dominant patterns are sessions and JWT tokens.

Sessions vs JWT

  • Session (stateful): server stores session, client holds a session-id cookie. Easy to revoke; needs shared store (Redis) when scaled.
  • JWT (stateless): signed token holds the claims; server verifies the signature, stores nothing. Scales horizontally; harder to revoke before expiry.

JWT structure

Three base64url parts: header.payload.signature. The signature (HMAC with a secret, or RSA/EC private key) proves integrity. The payload is NOT encrypted — anyone can read it; never put secrets in it.

Access + refresh tokens

  • Access token — short-lived (e.g. 15 min), sent on every request.
  • Refresh token — long-lived, stored securely (httpOnly cookie / DB), used to mint new access tokens. Can be revoked.
Storage: Prefer httpOnly, Secure, SameSite cookies over localStorage (which is XSS-readable). Always validate exp, iss, aud; use a strong secret/keys.
Libraries: jsonwebtoken for tokens, bcrypt/argon2 for password hashing, passport for strategies (OAuth, local, JWT).

🎯 Interview questions

Session vs JWT trade-offs?

Sessions: stateful, easy revoke, needs store. JWT: stateless, scales, but revocation is hard before expiry (need a blocklist or short TTL + refresh).

Is JWT payload encrypted?

No — only signed. It's base64, readable by anyone. Don't store secrets; use JWE if you need encryption.

How do you revoke a JWT?

Short-lived access tokens + revocable refresh tokens, or a server-side token blocklist/version check.

Where to store the token client-side?

httpOnly Secure SameSite cookie (mitigates XSS theft); pair with CSRF protection. Avoid localStorage for sensitive tokens.
