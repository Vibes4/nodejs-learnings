REST API design Express

Designing clean REST APIs is core senior territory. REST = resources identified by URLs, manipulated with HTTP verbs, communicated via status codes.

Verbs & semantics

  • GET /users — list · /users/:id — read. Safe & idempotent.
  • POST /users — create. Not idempotent.
  • PUT /users/:id — full replace. Idempotent.
  • PATCH /users/:id — partial update.
  • DELETE /users/:id — remove. Idempotent.

Status codes that matter

  • 200 OK · 201 Created · 204 No Content (delete).
  • 400 Bad Request · 401 Unauthorized · 403 Forbidden · 404 Not Found.
  • 409 Conflict · 422 Unprocessable · 429 Too Many Requests.
  • 500 Server Error · 503 Unavailable.

Design principles

  • Nouns, not verbs, in URLs (/users, not /getUsers).
  • Plural resource names; nest relationships (/users/:id/orders).
  • Versioning: /api/v1/... or header-based.
  • Pagination (?page=&limit= or cursor), filtering, sorting.
  • Consistent error shape; use idempotency keys for safe retries on POST.
  • HATEOAS / consistent envelopes are nice-to-have.
Idempotency: PUT/DELETE can be retried safely (same result). POST can't by default — clients use an Idempotency-Key header so retries don't double-charge.

🎯 Interview questions

PUT vs PATCH?

PUT replaces the entire resource (idempotent); PATCH applies a partial change.

Which methods are idempotent?

GET, PUT, DELETE, HEAD (same effect if repeated). POST and PATCH generally are not.

How do you version an API?

URL prefix (/v1), custom header, or content negotiation. URL prefix is most common and explicit.

201 vs 200 vs 204?

201 after creating (with Location/body), 200 for successful read/update with body, 204 for success with no body (e.g. DELETE).
