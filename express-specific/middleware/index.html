Middleware — the heart of Express Express

A middleware is a function (req, res, next) that runs during the request/response cycle. Express is essentially a pipeline of middleware. Mastering this is the #1 Express interview topic.

The signature

function mw(req, res, next) {
  // 1. inspect/modify req or res
  // 2. either end the response (res.send) OR call next()
  next();           // pass control to the next middleware
}

Each middleware must either send a response or call next(). Forgetting both hangs the request forever.

Types of middleware

  • Application-level — app.use(fn) (runs for all routes) or app.use('/api', fn).
  • Router-level — bound to an express.Router().
  • Route-level — passed directly to a route handler.
  • Built-in — express.json(), express.urlencoded(), express.static().
  • Third-party — cors, helmet, morgan, cookie-parser.
  • Error-handling — 4 args: (err, req, res, next).

Order is everything

Middleware runs in the order registered. Body parsers and auth must come before the routes that need them. Error handlers go last.

Pattern: Use middleware for cross-cutting concerns — logging, auth, validation, rate limiting, CORS — so route handlers stay focused on business logic.

🎯 Interview questions

What are the three arguments of middleware?

req, res, next. Error middleware has a fourth, err, first: (err, req, res, next).

What happens if you don't call next()?

The request hangs (no further middleware runs and no response is sent) — unless this middleware already sent a response.

How does Express know a middleware is an error handler?

By its arity — it has exactly 4 parameters. That's how Express distinguishes it.

Where do you put body parsers and auth?

Before the routes that depend on them; error handlers go after all routes.
