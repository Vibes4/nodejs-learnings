Transactions & Consistency MongoDB

A single-document write in MongoDB is always atomic. For multi-document atomicity you need explicit multi-document transactions (since 4.0 on replica sets, 4.2 on sharded clusters).

Using transactions (Mongoose)

const session = await mongoose.startSession();
try {
  session.startTransaction();
  await Account.updateOne({ _id: a }, { $inc: { balance: -100 } }, { session });
  await Account.updateOne({ _id: b }, { $inc: { balance:  100 } }, { session });
  await session.commitTransaction();   // all-or-nothing
} catch (e) {
  await session.abortTransaction();    // rollback
} finally {
  session.endSession();
}

Design first — transactions second

MongoDB's philosophy: model so related data lives in one document and you rarely need transactions. Reach for them on genuine cross-document invariants (money transfers, inventory + order).

Consistency knobs

  • Write concern (w) — how many nodes must ack a write (w:'majority' for durability).
  • Read concern — local, majority, linearizable — controls how fresh/consistent reads are.
  • Read preference — primary vs secondary (stale reads on secondaries).
Cost: Multi-doc transactions add latency and locking, can abort under contention (retry with backoff), and have time/size limits. Don't use them as a substitute for good schema design.

🎯 Interview questions

Are MongoDB writes atomic?

Yes at the single-document level (including embedded sub-docs). Across documents you need a multi-document transaction.

What infra do transactions require?

A replica set (or sharded cluster on 4.2+). They don't work on a standalone mongod.

What is write concern majority?

The write is acknowledged only after a majority of replica-set members persist it — guarantees durability against failover.

How does Mongo's model reduce the need for transactions?

By embedding related data in one document, atomic single-doc updates cover most consistency needs — transactions become the exception.
