Indexing & Performance MongoDB

Without an index, a query does a collection scan (COLLSCAN) — reading every document. Indexes are B-trees that make lookups, sorts, and ranges fast. This is the #1 MongoDB performance topic.

Index types

  • Single-field — { email: 1 }.
  • Compound — { status: 1, createdAt: -1 } (order matters).
  • Multikey — automatically on array fields.
  • Text, Geospatial (2dsphere), Hashed (sharding), TTL (auto-expire docs), Partial / Sparse, Unique.

The ESR rule for compound indexes

Order fields as Equality → Sort → Range. A compound index also serves queries on any left-to-right prefix of its keys.

// index { status: 1, createdAt: -1, amount: 1 } supports:
//   {status}              ✅ prefix
//   {status, createdAt}   ✅ prefix
//   {createdAt}           ❌ not a prefix

Reading .explain()

  • IXSCAN = index used 🎉; COLLSCAN = full scan 🚨.
  • Compare totalDocsExamined vs nReturned — close is good; huge ratio means a bad/missing index.
  • Covered query: all needed fields are in the index → no document fetch at all.
Indexes aren't free: every write must update every index, and they consume RAM. Over-indexing slows writes and wastes memory. Index for real query patterns, then drop unused ones.

🎯 Interview questions

IXSCAN vs COLLSCAN?

IXSCAN uses an index (fast); COLLSCAN reads every doc (slow). explain('executionStats') reveals which.

Explain the ESR rule.

In a compound index, put Equality fields first, then the Sort field, then Range fields — maximizes index usage and avoids in-memory sorts.

What is a covered query?

A query answered entirely from the index (projection + filter all indexed), so MongoDB never reads the documents.

Downside of too many indexes?

Slower writes (each index updated), more RAM/storage. Index deliberately for query patterns.

What's a TTL index for?

Auto-expiring documents after a time (sessions, logs, OTPs) via a background deletion task.
