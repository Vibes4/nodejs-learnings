Aggregation Pipeline MongoDB

The aggregation pipeline transforms documents through a sequence of stages, each feeding the next — MongoDB's answer to SQL GROUP BY, joins, and analytics.

Common stages (order matters)

  • $match — filter (like WHERE). Put it first to use indexes and shrink the dataset early.
  • $group — aggregate by a key with accumulators ($sum, $avg, $max, $push).
  • $project — reshape/select fields (like SELECT).
  • $sort / $limit / $skip — order & paginate.
  • $lookup — left outer join to another collection.
  • $unwind — flatten an array into one doc per element.
  • $facet — run multiple sub-pipelines in one pass (e.g. data + count).

Performance rules

  • $match and $sort early so they can use indexes (only the first stages can).
  • $project to drop big unused fields before heavy stages.
  • Watch the 100MB per-stage memory limit — use allowDiskUse or index-backed sorts.
  • Verify with .explain(); $lookup on unindexed foreign fields is slow.
SQL ↔ Mongo: WHERE→$match, GROUP BY→$group, SELECT→$project, JOIN→$lookup, ORDER BY→$sort, HAVING→$match after $group.

🎯 Interview questions

Why put $match first?

Early filtering reduces documents flowing downstream and lets the stage use indexes — only leading stages can leverage an index.

$group accumulators you'd name?

$sum, $avg, $min/$max, $push, $addToSet, $first/$last.

How do you join collections?

$lookup (left outer join); index the foreign field. Follow with $unwind if you want one row per match.

aggregate vs find?

find is for simple filtered reads; aggregation does multi-stage transforms, grouping, joins, and computed fields.
