Transactions & ACID SQL

Transactions group statements into an all-or-nothing unit. ACID is the guarantee set relational databases provide — and a near-certain interview question.

ACID

  • Atomicity — all statements succeed or none do (COMMIT / ROLLBACK).
  • Consistency — the DB moves from one valid state to another (constraints, FKs hold).
  • Isolation — concurrent transactions don't corrupt each other.
  • Durability — once committed, it survives crashes (WAL / fsync).

  • Dirty read — read uncommitted data.
  • Non-repeatable read — same row, different value on re-read.
  • Phantom read — same query, new rows appear.

Locking & concurrency

  • Pessimistic — SELECT ... FOR UPDATE locks rows up front.
  • Optimistic — a version column; check-and-retry on conflict.
  • Deadlocks — two txns wait on each other; the DB kills one. Avoid by locking in a consistent order.
Always release the connection. Run the whole transaction on one pooled client and release() it in finally, even on error — leaked connections exhaust the pool.

🎯 Interview questions

Explain ACID.

Atomicity (all-or-nothing), Consistency (valid states only), Isolation (concurrent safety), Durability (survives crashes).

Dirty vs non-repeatable vs phantom read?

Dirty = reading uncommitted data; non-repeatable = a row's value changes between reads; phantom = new rows match a re-run query.

Default isolation level?

Postgres: Read Committed. MySQL/InnoDB: Repeatable Read. Serializable is strongest but costs concurrency.

Optimistic vs pessimistic locking?

Pessimistic locks rows during the txn (FOR UPDATE); optimistic detects conflicts via a version and retries — better under low contention.

What causes a deadlock and how do you prevent it?

Transactions acquiring locks in opposite orders. Prevent by always locking resources in a consistent order and keeping transactions short.
