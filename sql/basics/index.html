SQL (Postgres / MySQL) DB

Relational databases store normalized data in tables with enforced schemas and relationships. In Node you use a driver (pg, mysql2), a query builder (Knex), or an ORM (Prisma, Sequelize, TypeORM).

Connection pooling (critical)

Opening a connection per request is slow and exhausts the DB. Always use a pool — a fixed set of reusable connections. Tune pool size to your DB's max_connections and worker count.

Parameterized queries — always

// SAFE — values are bound, not concatenated
pool.query('SELECT * FROM users WHERE email = $1', [email]);

// VULNERABLE — never do this (SQL injection)
pool.query(`SELECT * FROM users WHERE email = '${email}'`);

Joins, transactions, indexes

  • JOINs — INNER (matches only), LEFT (all left + matched right), etc.
  • Transactions — BEGIN … COMMIT/ROLLBACK for atomic multi-statement writes (ACID). Use one connection from the pool for the whole transaction.
  • Indexes — B-tree on filtered/joined/sorted columns; verify with EXPLAIN ANALYZE. Beware over-indexing (slows writes).
  • Isolation levels — Read Committed (default in PG) → Serializable; understand dirty/non-repeatable/phantom reads.

SQL vs NoSQL (the classic)

  • SQL — strong schema, relations, ACID, complex joins/reporting. Vertical scaling first.
  • NoSQL — flexible schema, horizontal scale, denormalized, eventual consistency options.
  • Choose by data shape & access patterns, not hype. Many systems use both.
N+1 query problem: fetching a list then querying per-row in a loop. Fix with a JOIN or a single WHERE id IN (...). ORMs hide this — watch the logs.

🎯 Interview questions

Why connection pooling?

Connections are expensive to establish and DBs cap them. A pool reuses a bounded set, cutting latency and preventing exhaustion.

How do you prevent SQL injection?

Parameterized/prepared statements (bound values), never string interpolation. ORMs/builders do this by default.

What are ACID and isolation levels?

Atomicity, Consistency, Isolation, Durability. Isolation levels (Read Uncommitted → Serializable) trade consistency for concurrency, preventing dirty/non-repeatable/phantom reads.

INNER vs LEFT JOIN?

INNER returns only matching rows; LEFT returns all left rows with NULLs where no right match exists.

When SQL vs NoSQL?

SQL for relational, transactional, report-heavy data with a stable schema; NoSQL for flexible/denormalized data needing horizontal scale and high write throughput.

How do you debug a slow query?

EXPLAIN ANALYZE — look for seq scans on big tables, missing indexes, bad join order, high row estimates vs actual.
