Indexing & Query Tuning SQL

Indexes are the difference between a 2ms and a 2s query. The default is a B-tree index; understanding when it's used (and when it isn't) is core senior knowledge.

Index types

  • B-tree — default; equality, range, sorting, prefix matches.
  • Hash — equality only.
  • Composite — multiple columns; leftmost-prefix rule applies.
  • Partial — index a subset (WHERE active = true).
  • Covering — includes all columns a query needs → index-only scan.
  • Postgres extras: GIN/GiST (JSONB, full-text, arrays), BRIN (huge ordered tables).

When an index is NOT used

  • Function/transform on the column: WHERE lower(email)=... (fix with a functional index).
  • Leading wildcard: LIKE '%foo'.
  • Implicit type cast, or low selectivity (planner decides a scan is cheaper).
  • Composite index but query skips the leftmost column.

EXPLAIN ANALYZE

  • Index Scan / Index Only Scan = good; Seq Scan on a big table = investigate.
  • Compare estimated vs actual rows — large gaps mean stale stats (ANALYZE).
  • Look for expensive Sort/Hash spilling to disk.
Trade-off: every index speeds reads but slows INSERT/UPDATE/DELETE and uses storage. Index the columns you actually filter, join, and sort on — then drop the unused ones.

🎯 Interview questions

Why isn't my index being used?

A function on the column, leading wildcard, type mismatch, low selectivity, or skipping the leftmost composite column. EXPLAIN shows the chosen plan.

What is a covering index?

An index containing every column the query needs, so the engine never touches the table (index-only scan).

Leftmost-prefix rule?

A composite index on (a,b,c) serves filters on a, a+b, a+b+c — but not b or c alone.

How do you debug a slow query?

EXPLAIN ANALYZE → find seq scans / bad estimates / disk sorts → add/adjust indexes, rewrite the query, update stats.
