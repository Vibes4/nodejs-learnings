Normalization & Schema Design SQL

Normalization organizes tables to eliminate redundancy and update anomalies. You should be able to explain the normal forms and, crucially, when to denormalize.

Normal forms

  • 1NF — atomic values, no repeating groups/arrays in a column; each row unique.
  • 2NF — 1NF + no partial dependency (non-key columns depend on the whole composite key).
  • 3NF — 2NF + no transitive dependency (non-key columns depend only on the key, not on other non-key columns).
  • BCNF — stricter 3NF; every determinant is a candidate key.

Anomalies normalization prevents

  • Update anomaly — change a value in many duplicated rows; miss one → inconsistent data.
  • Insert anomaly — can't add data without unrelated data.
  • Delete anomaly — deleting a row loses unrelated facts.

When to denormalize

  • Read-heavy reporting/analytics where joins are too slow.
  • Precomputed aggregates / materialized views / caching counts.
  • Accept controlled redundancy for performance — but guard consistency (triggers, jobs).
Constraints enforce consistency: PRIMARY KEY, FOREIGN KEY (referential integrity), UNIQUE, NOT NULL, CHECK. They're the relational model's correctness backbone — interviewers expect you to reach for them, not enforce everything in app code.

🎯 Interview questions

Explain 1NF, 2NF, 3NF quickly.

1NF: atomic columns, unique rows. 2NF: no partial dependency on part of a composite key. 3NF: no transitive dependency between non-key columns.

Why normalize?

To remove redundancy and prevent insert/update/delete anomalies — one fact stored in one place.

When would you denormalize?

For read/reporting performance: precomputed aggregates, fewer joins, materialized views — trading redundancy for speed, with consistency safeguards.

How is referential integrity enforced?

FOREIGN KEY constraints (with ON DELETE/UPDATE actions) plus UNIQUE/NOT NULL/CHECK — at the DB level, not just the app.
