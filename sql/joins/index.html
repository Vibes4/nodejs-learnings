Joins SQL

Joins combine rows from multiple tables on a related column — the core of the relational model. Knowing each join's exact result set is a guaranteed interview question.

Join types

  • INNER JOIN — only rows with a match in both tables.
  • LEFT (OUTER) JOIN — all left rows; right columns are NULL when no match.
  • RIGHT JOIN — mirror of LEFT (all right rows).
  • FULL OUTER JOIN — all rows from both; NULLs where no match.
  • CROSS JOIN — Cartesian product (every combination).
  • SELF JOIN — a table joined to itself (e.g. employee → manager).

Example

SELECT u.name, o.total
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE o.total > 50 OR o.total IS NULL;

Senior gotchas

  • LEFT JOIN + WHERE on the right table silently turns it into an INNER JOIN (NULLs fail the predicate). Put right-table conditions in the ON clause instead.
  • Always join on indexed columns (usually FK → PK) or it's a slow scan.
  • Fan-out: joining one-to-many duplicates the "one" side's rows; use DISTINCT or aggregate carefully.
  • Join order/strategy (nested loop, hash, merge) is chosen by the planner — check EXPLAIN.
N+1 vs JOIN: fetching a list then querying related rows per item (common with naive ORMs) is the N+1 problem. One JOIN (or WHERE id IN (...)) replaces N queries.

🎯 Interview questions

INNER vs LEFT JOIN?

INNER returns only matching rows; LEFT returns all left rows with NULLs where the right side has no match.

Why did my LEFT JOIN behave like an INNER JOIN?

A WHERE filter on the right table excludes the NULL rows. Move that condition into the ON clause to preserve unmatched left rows.

What is a self join?

Joining a table to itself with aliases — e.g. employees to their managers in the same table.

How does the DB execute a join?

The planner picks nested-loop, hash, or merge join based on indexes and row estimates; inspect with EXPLAIN ANALYZE.
