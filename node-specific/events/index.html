EventEmitter Node Core

The events module powers Node's pub/sub backbone. Streams, HTTP servers, sockets — almost every async API in Node is an EventEmitter under the hood.

Core API

  • emitter.on(event, listener) — subscribe (alias addListener).
  • emitter.once(event, listener) — fire at most once.
  • emitter.emit(event, ...args) — publish synchronously to all listeners.
  • emitter.off(event, listener) — unsubscribe (alias removeListener).
  • emitter.removeAllListeners(event), emitter.listenerCount(event).
Emit is synchronous. Listeners run in registration order, on the same tick. To defer, wrap with setImmediate.

The 'error' event is special

If an EventEmitter emits 'error' and there is no listener, Node throws and crashes the process. Always attach an error listener to emitters that can fail.

Memory leak warning

Adding more than 10 listeners for one event triggers a MaxListenersExceededWarning. It's usually a bug (you forgot to remove listeners). Tune with emitter.setMaxListeners(n) only when intentional.

Senior tip: Prefer once + cleanup, or AbortController / events.once(emitter, evt) (promise-based) to avoid leak-prone manual subscriptions.

🎯 Interview questions

Is emit sync or async?

Synchronous — all listeners run before emit returns, in the order added.

What happens if 'error' is emitted with no listener?

The error is thrown and the process crashes (uncaught). Always handle 'error'.

How do you avoid listener memory leaks?

Remove listeners when done, use once, use AbortSignal, and watch for the MaxListeners warning.

How is EventEmitter different from a Promise?

Emitter = multiple values over time, many listeners. Promise = single resolution. EventEmitter ≈ observable stream; Promise ≈ one-shot.
