Globals & process Node Core

Unlike the browser (where the global object is window), Node's global object is global (or the standard globalThis). Node also injects per-file "pseudo-globals" that are actually arguments to the module wrapper function.

The module wrapper

Every file you write is secretly wrapped by Node like this before execution:

(function (exports, require, module, __filename, __dirname) {
  // your code here
});

That is why require, module, exports, __dirname, and __filename are available even though you never declared them — and why they are not truly global.

Key globals

  • process — the running Node process: env vars, args, stdin/stdout, lifecycle.
  • __dirname / __filename — absolute path of the current dir/file (CommonJS only).
  • global / globalThis — the global namespace.
  • Buffer, setTimeout, setImmediate, queueMicrotask, console.

process essentials

  • process.env — environment variables (config, secrets).
  • process.argv — CLI arguments array.
  • process.cwd() — current working directory (where you ran node, ≠ __dirname).
  • process.exit(code) — terminate (0 = success).
  • process.on('SIGTERM' | 'uncaughtException' | 'unhandledRejection', cb) — graceful shutdown / safety nets.
Senior tip: Never let uncaughtException keep the process alive forever — log, flush, and exit. The process is in an undefined state. Use it only for graceful shutdown, not as a try/catch.

🎯 Interview questions

Is __dirname a true global?

No. It's a parameter of the module wrapper function, scoped per file. In ES modules it doesn't exist — you derive it from import.meta.url.

Difference between process.cwd() and __dirname?

__dirname is where the file lives; process.cwd() is where the process was launched from. They differ when you run a file from another directory.

How do you do graceful shutdown?

Listen for SIGTERM/SIGINT, stop accepting new connections (server.close()), drain in-flight requests, close DB pools, then process.exit(0) with a timeout fallback.
