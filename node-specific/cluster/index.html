Cluster Node Core

A single Node process uses one CPU core. The cluster module forks multiple worker processes that share the same server port, letting you use all cores for a network server.

How it works

  • The primary (master) process forks N workers (usually one per CPU core).
  • Each worker is a full Node process with its own memory and event loop.
  • The OS / primary load-balances incoming connections across workers (round-robin on most platforms).
  • Workers don't share memory — coordinate via IPC or external stores (Redis).

Resilience

Listen for 'exit' on the primary and re-fork dead workers — gives you zero-downtime crash recovery.

In practice: Most teams use a process manager like PM2 (pm2 start app.js -i max) or run multiple containers behind a load balancer instead of hand-writing cluster code — but you must understand the underlying model.

cluster vs worker_threads

  • cluster — multiple processes, separate memory, scales a network server across cores.
  • worker_threads — multiple threads in one process, shared memory, for CPU-bound computation.

🎯 Interview questions

How do cluster workers share a port?

The primary creates the listening socket and distributes connections (or the OS does, with SO_REUSEPORT). Workers accept handed-off connections.

Do cluster workers share memory?

No — separate processes. Share state via Redis, a DB, or IPC messages. Don't keep session state in worker memory.

cluster vs PM2 vs Kubernetes?

Same idea at different layers: cluster = in-app, PM2 = process manager, K8s = container replicas. Pick based on ops maturity; the scaling principle is identical.
