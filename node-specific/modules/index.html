Modules: CommonJS vs ESM Node Core

Node has two module systems. Knowing the differences cold is a classic senior question.

CommonJS (CJS) — the original

  • Load with require(), export with module.exports / exports.
  • Synchronous loading, resolved at runtime.
  • require() can be called conditionally, anywhere.
  • Default for .js files (unless "type":"module" in package.json).

ES Modules (ESM) — the standard

  • import / export, used by .mjs or with "type":"module".
  • Asynchronous, statically analyzed (imports hoisted, must be top-level).
  • Supports top-level await and tree-shaking.
  • No __dirname/require — use import.meta.url and createRequire.

module.exports vs exports

exports is just a reference to module.exports. Reassigning exports = ... breaks the link; only module.exports = ... actually changes what's exported.

Gotcha: exports = { foo } exports nothing. Use module.exports = { foo } or exports.foo = ....

Module caching

require() caches modules by resolved path. The module body runs once; subsequent requires return the cached module.exports. This makes modules effective singletons — great for DB connections, dangerous for stateful surprises.

🎯 Interview questions

CJS vs ESM in one line each?

CJS = synchronous, dynamic, runtime-resolved (require). ESM = async, static, hoisted, standardized (import), enables tree-shaking and top-level await.

Why are modules singletons?

The module cache stores module.exports by resolved path; the file body executes only on first require.

Can you require an ESM module from CJS?

Not with require() directly (older Node). Use dynamic await import(), which works from both systems.

How does Node resolve require('x')?

Core module → relative/absolute path → walk up node_modules folders → check package.json "main"/"exports" → index.js.
