HTTP (no framework) Node Core

The http module is what Express is built on. Understanding raw http shows you exactly what Express abstracts away — a favorite senior interview angle ("what does Express actually add?").

Creating a server

http.createServer((req, res) => {...}) returns a server; the callback runs for every request. req is a Readable stream (IncomingMessage); res is a Writable stream (ServerResponse).

What you must do BY HAND without Express

  • Routing — parse req.url + req.method with if/switch.
  • Body parsing — manually collect 'data' chunks and JSON.parse on 'end'.
  • Query/params — parse with the URL class yourself.
  • Headers / status — res.writeHead(status, headers), res.setHeader.
  • Send response — res.end(body) (you must stringify and set Content-Type).
This is the whole point: Express turns all of the above into one-liners (app.get, res.json, express.json(), middleware). See the "Why Express over Node?" page.

Making requests

http.request() / http.get() act as a client. Modern code often uses the global fetch() (stable since Node 18).

🎯 Interview questions

What is req / res really?

req = IncomingMessage, a Readable stream. res = ServerResponse, a Writable stream. That's why you read the body via stream events.

How do you read a POST body in raw http?

Accumulate req.on('data', chunk => ...) into a buffer/array, then on req.on('end') concat and parse. Guard against oversized bodies (DoS).

Does createServer spawn a thread per request?

No — one event loop handles all connections via non-blocking I/O. That's Node's core scalability model vs thread-per-request servers.
