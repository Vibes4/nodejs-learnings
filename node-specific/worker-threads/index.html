Worker Threads Node Core

worker_threads run JavaScript on real OS threads inside the same process, sharing memory. They're the right tool for CPU-bound work (image processing, parsing, encryption, big calculations) that would otherwise block the event loop.

Why they exist

Node's event loop is single-threaded for JS. A heavy synchronous computation freezes all requests. Worker threads move that computation off the main thread so the event loop stays responsive.

Key pieces

  • new Worker(file, { workerData }) — start a thread.
  • parentPort.postMessage() / worker.on('message') — message passing (structured clone).
  • workerData — initial data passed into the worker.
  • SharedArrayBuffer + Atomics — true shared memory (no copy), for high-performance cases.

worker_threads vs child_process vs cluster

  • worker_threads — threads, shared memory, CPU-bound JS work.
  • child_process — separate processes, run any program, heavier isolation.
  • cluster — processes that share a server socket to scale a network server.
Rule of thumb: I/O-bound → stay on the event loop (async). CPU-bound → worker threads (or a job queue). Scale a web server across cores → cluster / PM2 / containers.

🎯 Interview questions

When do you use worker threads?

For CPU-bound tasks that block the loop. I/O is already non-blocking, so threads don't help there.

Do worker threads share memory?

Messages are copied (structured clone) by default. For true sharing use SharedArrayBuffer + Atomics.

Threads vs cluster for an HTTP API?

To use all cores for request handling, cluster (separate processes sharing the port). Worker threads are for offloading specific heavy computations within a process.
