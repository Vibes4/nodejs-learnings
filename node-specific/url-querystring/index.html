URL & Query String Node Core

Parsing URLs and query strings correctly matters for routing, redirects, and security. The modern WHATWG URL class replaces the legacy url.parse().

The URL class (preferred)

  • new URL(input, base?) — parses into protocol, host, hostname, port, pathname, search, hash.
  • url.searchParams — a URLSearchParams object: .get(), .getAll(), .set(), .append(), iterate.

URLSearchParams

Handles encoding/decoding automatically, supports repeated keys (?a=1&a=2 → getAll('a') = ['1','2']), and serializes back with toString().

Legacy querystring + url.parse() are still around but discouraged: url.parse has known parsing quirks/security issues. Prefer the URL class.

🎯 Interview questions

Legacy url.parse vs WHATWG URL?

The URL class is standards-compliant (same as browsers), safer, and gives you searchParams. url.parse is legacy with edge-case bugs.

How do repeated query keys behave?

searchParams.get returns the first; getAll returns all values as an array.

Why not build query strings by concatenation?

You must URL-encode values (spaces, &, =). URLSearchParams encodes for you, preventing injection/breakage.
