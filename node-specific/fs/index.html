File System (fs) Node Core

The fs module reads/writes files and directories. It offers three flavors of every operation — knowing when to use each is the interview point.

Three APIs

  • Synchronous — fs.readFileSync(). Blocks the event loop. OK at startup / CLI scripts, never in a request handler.
  • Callback — fs.readFile(path, cb). Classic error-first callbacks.
  • Promise — require('fs/promises') or fs.promises. Modern, works with async/await. Preferred.

Common operations

  • Read/write: readFile, writeFile, appendFile.
  • Metadata: stat / lstat (size, mtime, isDirectory).
  • Dirs: mkdir({recursive:true}), readdir, rm({recursive:true}).
  • Existence: prefer fs.access() or just try the op and catch — avoid existsSync race conditions (TOCTOU).
  • Watching: fs.watch() for change events.
Big files: readFile loads the entire file into memory. For large files use streams (fs.createReadStream) to process in chunks.
Always pass an encoding (e.g. 'utf8') or you get a Buffer back instead of a string.

🎯 Interview questions

When is readFileSync acceptable?

At startup (loading config), in build scripts / CLIs — anywhere not on the hot request path. It blocks the single event-loop thread.

readFile vs createReadStream?

readFile buffers the whole file in memory; streams process chunk-by-chunk with backpressure — essential for large files and constant memory.

Why avoid existsSync before reading?

Time-of-check-to-time-of-use race: the file can be deleted between the check and the read. Just attempt the operation and handle ENOENT.
