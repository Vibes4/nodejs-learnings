The Event Loop & Timers Node Core

The single most important senior topic. Node is single-threaded for your JS, but uses libuv + a thread pool for I/O. The event loop is what lets one thread handle thousands of concurrent connections.

Event loop phases (in order)

  1. timers — setTimeout, setInterval callbacks.
  2. pending callbacks — deferred I/O callbacks.
  3. poll — retrieve new I/O events; execute I/O callbacks (most work happens here).
  4. check — setImmediate callbacks.
  5. close callbacks — e.g. socket.on('close').

Microtasks run BETWEEN phases

After every macrotask (and between phases), Node drains the microtask queues in this priority:

  1. process.nextTick() queue (highest priority — runs before promises).
  2. Promise microtask queue (.then, await, queueMicrotask).
Starvation: Recursive process.nextTick can starve the event loop entirely — the loop never advances to the next phase. Prefer setImmediate for "run after I/O".

setTimeout(0) vs setImmediate

  • Inside an I/O callback, setImmediate always fires before setTimeout(0) (check phase comes right after poll).
  • At the top level, the order is non-deterministic (depends on process startup timing).
Senior tip: CPU-bound work blocks the event loop and kills throughput. Offload to worker_threads, the C++ thread pool, or another service. "Don't block the loop" is the golden rule.

🎯 Interview questions

Is Node single-threaded?

Your JavaScript runs on one thread, but libuv maintains a thread pool (default 4) for fs, DNS, crypto, zlib. So Node is "single-threaded event loop + background thread pool".

nextTick vs Promise vs setImmediate ordering?

nextTick → Promises (both microtasks, drained before continuing) → setImmediate (check phase). Timers vs immediate depends on context.

What blocks the event loop?

Synchronous CPU work (big loops, JSON.parse of huge payloads, sync crypto, fs.*Sync), and recursive nextTick.

How do you size the libuv thread pool?

UV_THREADPOOL_SIZE env var (max 1024). Raise it if fs/crypto-heavy.
