Crypto Node Core

The crypto module provides hashing, HMAC, symmetric/asymmetric encryption, and secure random values. Critical for any auth-related interview.

Hashing (one-way)

  • crypto.createHash('sha256').update(data).digest('hex') — fast, deterministic, irreversible.
  • Used for checksums and integrity — NOT for passwords directly.

Passwords: never plain SHA

SHA is too fast → brute-forceable. Use a slow, salted KDF:

  • crypto.scrypt / crypto.pbkdf2 (built-in), or bcrypt / argon2 (libraries).
  • Each password gets a unique random salt to defeat rainbow tables.

Other essentials

  • crypto.randomBytes(n) / crypto.randomUUID() — secure tokens/IDs.
  • crypto.createHmac(algo, key) — keyed hash for signing (e.g. webhooks, JWT HS256).
  • crypto.timingSafeEqual(a, b) — constant-time compare to prevent timing attacks.
  • AES via createCipheriv/createDecipheriv for symmetric encryption.
Never use Math.random() for tokens, session IDs, or salts — it's not cryptographically secure. Use crypto.randomBytes/randomUUID.

🎯 Interview questions

Why not hash passwords with SHA-256?

SHA is designed to be fast → billions of guesses/sec. Passwords need a deliberately slow, salted algorithm (bcrypt/scrypt/argon2) with a per-user salt.

Hashing vs encryption?

Hashing is one-way (can't reverse) — for integrity/passwords. Encryption is two-way (decrypt with a key) — for confidential data you need back.

Why timingSafeEqual?

=== short-circuits, leaking length/position info via timing. Constant-time comparison prevents an attacker from inferring secrets.

What's a salt and why per-user?

Random data mixed into the hash so identical passwords hash differently, defeating precomputed rainbow tables.
