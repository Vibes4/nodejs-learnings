Streams Node Core

Streams process data incrementally instead of loading everything into memory. They are the foundation of Node's I/O performance — HTTP requests/responses, file I/O, and compression are all streams.

Four stream types

  • Readable — source you read from (fs.createReadStream, req).
  • Writable — sink you write to (fs.createWriteStream, res).
  • Duplex — both (TCP socket).
  • Transform — duplex that modifies data (zlib.createGzip, crypto).

Modes of reading

  • Flowing — data pushed via 'data' events.
  • Paused — you pull with .read(); for await...of is the modern way.

pipe & backpressure

readable.pipe(writable) connects them and automatically handles backpressure: if the writable is slower than the readable, pipe pauses the source until the sink drains. This is the killer feature — constant memory regardless of data size.

Use pipeline(), not .pipe(): stream.pipeline(src, transform, dest, cb) propagates errors and cleans up file descriptors. Raw .pipe() leaks on errors.
highWaterMark sets the internal buffer size (default 16KB for files, 16 objects in object mode). It's the threshold that triggers backpressure.

🎯 Interview questions

What problem do streams solve?

Processing data larger than RAM and reducing latency (start processing the first byte before the last arrives) — constant memory, better throughput.

What is backpressure?

When the consumer is slower than the producer. pipe/pipeline pause the source until drain, preventing unbounded memory growth.

pipe vs pipeline?

pipeline adds proper error propagation and resource cleanup across the whole chain; pipe does not forward errors, risking leaks.

Give a real example.

Streaming a large file to an HTTP response gzipped: pipeline(fs.createReadStream(f), zlib.createGzip(), res).
