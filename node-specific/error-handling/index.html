Error Handling in Node.js Node Core

Node has four distinct error channels depending on how the code is written. Using the wrong one (e.g. try/catch around an async callback) silently misses errors — a classic senior trap.

1. Synchronous errors → try/catch

try {
  const data = JSON.parse(badInput);   // throws synchronously
} catch (err) {
  console.error('handled:', err.message);
}

try/catch only catches synchronous throws. It will not catch errors thrown later inside a callback or an unawaited promise.

2. Error-first callbacks → (err, data)

fs.readFile('x.txt', (err, data) => {
  if (err) return handle(err);   // ALWAYS check err first
  use(data);
});

Node's core convention: the first callback argument is the error (or null). A try/catch around fs.readFile catches nothing — the error arrives in the callback.

3. Promises / async-await → .catch or try/catch

// promise chain
doAsync().then(use).catch(handle);

// async/await — try/catch DOES work here
try { const data = await doAsync(); }
catch (err) { handle(err); }

4. EventEmitter → the 'error' event

stream.on('error', (err) => handle(err));
// If an emitter emits 'error' with NO listener, Node throws & crashes.

Process-level safety nets (last resort)

  • process.on('uncaughtException', cb) — a synchronous error escaped everything.
  • process.on('unhandledRejection', cb) — a promise rejected with no .catch.
These are not a try/catch. After an uncaughtException the process is in an undefined state — log, flush, and exit, then let a process manager restart you. Don't resume normal work.

Custom error classes

class AppError extends Error {
  constructor(message, code) { super(message); this.name = 'AppError'; this.code = code; }
}
throw new AppError('not found', 404);

Distinguish operational errors (expected: bad input, network down — handle & continue) from programmer errors (bugs: undefined is not a function — let it crash).

🎯 Interview questions

Why doesn't try/catch catch an async error?

try/catch only wraps the synchronous call stack. By the time a callback or promise settles, the try block has already exited. Use error-first callbacks, .catch, or await inside try/catch.

What happens on an unhandled 'error' event?

If an EventEmitter emits 'error' and nothing is listening, Node throws it as an uncaught exception and the process crashes.

uncaughtException vs unhandledRejection?

The first is a thrown sync error nobody caught; the second is a rejected promise with no rejection handler. Both are last-resort hooks — log and exit, don't continue.

Operational vs programmer errors?

Operational = expected runtime failures you handle gracefully (retry, 4xx). Programmer = bugs you should fix; crashing + restarting is often the safest response.

Should you swallow errors in a catch?

Never silently. At minimum log with context; rethrow or translate to a typed error so callers can react.
