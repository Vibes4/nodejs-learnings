Child Process Node Core

The child_process module runs external commands and other Node scripts in separate OS processes — the way to leverage multiple cores or shell out to system tools.

Four ways to spawn

  • spawn(cmd, args) — streams stdout/stderr. Best for large/long output (no buffer limit). No shell by default.
  • exec(cmdString) — runs in a shell, buffers all output into a callback. Convenient but capped by maxBuffer (default ~1MB).
  • execFile(file, args) — like exec but no shell (safer, faster).
  • fork(modulePath) — special spawn for Node scripts with a built-in IPC channel (child.send / process.on('message')).

spawn vs exec — the classic question

  • spawn = streaming, unbounded, no shell → use for big output / long-running.
  • exec = buffered, shell, simple → use for short commands; risky with user input (shell injection).
Security: Never pass unsanitized user input to exec (runs in a shell → command injection). Prefer execFile/spawn with an args array and shell:false.
When to reach for it: CPU-bound work that would block the event loop, or wrapping CLI tools (ffmpeg, git). For sharing the same server port across cores, use cluster; for in-process CPU parallelism, worker_threads.

🎯 Interview questions

spawn vs exec vs fork?

spawn: streamed output, no buffer cap. exec: shell + buffered output. fork: spawn a Node module with an IPC channel for messaging.

How do parent and forked child communicate?

Over IPC: child.send(msg) / child.on('message') in the parent, and process.on('message') / process.send() in the child.

child_process vs worker_threads?

child_process = separate OS process (separate memory, heavier, can run any program). worker_threads = threads in the same process sharing memory via SharedArrayBuffer (lighter, JS only).
